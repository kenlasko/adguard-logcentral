# v0.2.3

A security-hardening patch on top of [v0.2.2](0.2.2.md): defense-in-depth
improvements to the HTTP layer from an OWASP-guided audit. No configuration or
deployment changes -- upgrade in place.

## What's hardened

- **Security response headers on every response.** The app now sends a strict
  `Content-Security-Policy` (`default-src 'self'`, `object-src 'none'`,
  `base-uri 'none'`, `frame-ancestors 'none'`), along with
  `X-Content-Type-Options: nosniff`, `X-Frame-Options: DENY`, and
  `Referrer-Policy: no-referrer`. Because the interface self-hosts its only
  script (htmx) and stylesheet, the CSP needs no inline exceptions. The
  `frame-ancestors`/`X-Frame-Options` directives block clickjacking of the
  Block / Unblock controls, and `no-referrer` keeps query-log URLs out of
  upstream `Referer` headers.
- **No caching of authenticated content.** Rendered pages and htmx fragments now
  carry `Cache-Control: no-store`, so query log data -- client addresses and the
  domains they resolved -- is never held in a browser or intermediary cache.
- **Tighter post-login redirect handling.** The `return` parameter used to send
  you back where you started after login now also rejects a leading `/\`, which
  browsers normalize to `//` and could otherwise be crafted into an off-site
  redirect. Only local, absolute paths are honored.
- **Validated response-status filter.** The query log status filter is now
  checked against the set of values AdGuard actually recognizes before being
  forwarded upstream; anything else falls back to "all responses" instead of
  passing unvalidated input through.

None of these change how you configure or operate the app; they close paper cuts
surfaced by the audit before they can matter.

## Container images

Multi-architecture images (`linux/amd64` and `linux/arm64`) are published to the
GitHub Container Registry:

```
ghcr.io/kenlasko/adguard-logcentral:0.2.3    # this release
ghcr.io/kenlasko/adguard-logcentral:0.2      # latest patch of the 0.2 line
ghcr.io/kenlasko/adguard-logcentral:latest   # newest release
```

Each image ships with a signed SBOM and build provenance attestation, is scanned
with Trivy before publication, and is signed with cosign keyless.

## Upgrading

Drop-in for v0.2.x: no new or changed environment variables, no schema, no
storage. Pull the new image (or move your `0.2` / `latest` tag) and restart.

See the [README](../../README.md) for full configuration, deployment
(docker run / compose / Kubernetes), and OIDC setup instructions.
