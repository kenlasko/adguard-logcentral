# v0.2.4

A security-hardening patch on top of [v0.2.3](0.2.3.md): two CodeQL findings
closed, one in the running app and one in the test harness. No configuration or
deployment changes -- upgrade in place.

## What's hardened

- **Log-injection-safe Block / Unblock logging.** When applying or removing a
  block rule fails, the handler logs the operator-supplied instance and domain.
  Those values are already validated before they reach the log call, but they
  are now also stripped of carriage returns and line feeds, so a crafted value
  can never forge additional log lines (CWE-117). This keeps each failure a
  single, trustworthy log record.
- **Registered-only redirects in the fake OIDC issuer.** The test-only fake
  identity provider used to exercise the login-to-callback flow now redirects
  solely to `redirect_uri` values a client has registered up front, mirroring
  how a real OpenID provider restricts redirects to pre-registered URIs. An
  unregistered value is rejected rather than followed, closing an open-redirect
  finding (CWE-601) in the test scaffolding.

Neither change affects how you configure or operate the app; they close static-
analysis findings before they can matter.

## Container images

Multi-architecture images (`linux/amd64` and `linux/arm64`) are published to the
GitHub Container Registry:

```
ghcr.io/kenlasko/adguard-logcentral:0.2.4    # this release
ghcr.io/kenlasko/adguard-logcentral:0.2      # latest patch of the 0.2 line
ghcr.io/kenlasko/adguard-logcentral:latest   # newest release
```

Each image ships with a signed SBOM and build provenance attestation, is scanned
with Trivy before publication, and is signed with cosign keyless.

## Upgrading

Drop-in for v0.2.x: no new or changed environment variables, no schema, no
storage. Pull the new image (or move your `0.2` / `latest` tag) and restart.

See the [README](../../README.md) for full configuration, deployment
(docker run / compose / Kubernetes), and OIDC setup instructions.
