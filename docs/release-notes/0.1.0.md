# v0.1.0

First release of **AdGuard Log Central** -- a single, central web UI that fetches
query logs **live** from multiple [AdGuard Home](https://adguard.com/en/adguard-home/overview.html)
instances, merges them in true time order, and presents one unified, filterable,
paginating view behind OIDC login. No database and no disk writes: every request
fans out to each instance on demand. Built for people running several AdGuard
Home instances for highly available DNS (for example kept in sync with
[adguardhome-sync](https://github.com/bakito/adguardhome-sync)) who are tired of
opening three web UIs to find where a DNS block happened.

## Highlights

- **Unified query log** across all configured instances, merged in true time
  order with correct cross-instance pagination (Load more / infinite scroll).
- **Filtering** by search term (domain or client), response status
  (blocked / rewritten / allowed), and per-instance color chips -- each instance
  gets its own color; click a chip to toggle it in or out of the view.
- **Block / unblock any domain** straight from a log row. Choose which instance
  the rule applies to (remembered in your browser via LocalStorage); with
  adguardhome-sync the change fans out to the rest.
- **Live tail** via an auto-refresh toggle, plus a manual refresh button.
- **Instance health bar** showing each instance's reachability, version, and
  protection state, refreshed every 15 seconds.
- **Aggregate stats page**: merged totals (queries, blocked, blocked share,
  average processing time) plus combined top-10 blocked domains, clients, and
  queried domains, each with a per-instance breakdown.
- **Partial-results handling**: if an instance is unreachable, the reachable
  instances still render and a banner names the ones that failed.
- **OIDC login only** (tested with [Pocket-ID](https://pocket-id.org/)), with
  stateless, encrypted (AES-256) session cookies.

## Design

- **No persistent storage.** Logs are fetched live from each instance on every
  request; there is no database and nothing is written to disk.
- **All configuration via environment variables and secrets.** Instances are
  declared with indexed variables (`ADGUARD_<n>_URL`, `_USERNAME`, `_PASSWORD`,
  optional `_NAME`) so each password maps cleanly to a single secret. Config is
  validated at startup and **all** problems are reported at once.
- **Server-rendered, no client build step.** The frontend is Go `html/template`
  plus vendored htmx and hand-written, dark-mode-aware CSS.
- **Unauthenticated `/healthz`** for liveness/readiness probes, reporting the
  running build version, commit, and date. Every other route requires a valid
  session.

## Container images

Multi-architecture images (`linux/amd64` and `linux/arm64`) are published to the
GitHub Container Registry:

```
ghcr.io/kenlasko/adguard-logcentral:0.1.0    # this release
ghcr.io/kenlasko/adguard-logcentral:0.1      # latest patch of the 0.1 line
ghcr.io/kenlasko/adguard-logcentral:latest   # newest release
```

Each image ships with a signed SBOM and build provenance attestation, is scanned
with Trivy before publication, and (for releases) is signed with cosign keyless.

## Requirements and notes

- Built on **Go 1.26**.
- OIDC is the only authentication method; a confidential client with the
  `openid`, `profile`, and `email` scopes is required.
- **Pagination trade-off:** cursors use each instance's raw nanosecond `time`
  value byte-for-byte. In the vanishingly rare case where a single instance emits
  two entries with the identical nanosecond timestamp that straddle a page
  boundary, one may be dropped from the view -- the deliberate cost of a
  stateless, no-storage design.
